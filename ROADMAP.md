# Roadmap

Живой список того, что хотим доделать. Не строгий план — приоритеты двигаем по
ситуации. Статусы выверены по коду (а не по комментариям), дата ревизии: 2026-06-17.

## 1.3 — подкатегории и порядок в счетах ✅

Версия **1.3.0**: большой релиз по пользовательским пожеланиям (issues #3–#15).
Главная тема — **подкатегории во всех видах**: круг «Доля категорий» с drill-in и
выносными подписями, стек-бары и treemap (с полноэкранным режимом), единые тултипы
с разбивкой по подкатегориям (#7); **каскадный фильтр по подкатегориям** с
независимым выбором категории/подкатегории (#9). Плюс: **«Потоки денег»** — узлы
«Сбережения» / «Из накоплений» (#8); **«Аномалии»** учитывают глобальные фильтры
(#6); **совокупный баланс** не уходит в минус (стартовые остатки по датам, #3) и
**подушка** учитывает забалансовые счета (#4); список счетов — таблица по
умолчанию, архив отдельной группой + «Скрыть архивные», фиксированные колонки;
**модалка отката** несинхронизированных изменений (#13); в карточке операции —
единое каскадное поле категории/подкатегории (#12), группировка счетов
(частые/активные/архивные) и время операции (#15); дубликаты по дате (#10).

Открытыми остаются: #5 (фиксированность vs обязательность — нужно решение по
модели), #11 (полная Wallet-страница: встроенные тренды + топ-N — drill-in уже
сделан), #14 (сквозной фильтр оборота).

## 1.2 — правило 50/30/20 ✅

Версия **1.2.0**: страница **«50/30/20»** — нужды/желания/сбережения против
ориентира + тренд нормы сбережений; чистая логика `lib/needsWants.ts` (+тесты).
Деление нужды/желания берётся из поля **`required`** («обязательная») у категории
в API Дзена — по умолчанию обязательно (`required !== false` → нужда). Добавлен
**редактор «обязательность расходов в
категориях»** на странице «Категории» (две кнопки + колонка «Тип категории» из
`showIncome`/`showOutcome`) с **синхронизацией поля в облако** (секция `tag` в
push-конвейере; `buildTagPush` + оверлей `useTagEditsStore`, +тесты). Плюс **единые цвета категорий** (из API Дзена / детерминированный фолбэк,
одинаково везде) и пачка UI-правок (фильтры «Выбрать/Снять все», тултипы только при
обрезке, и т.д.).

Страница «Прогноз кэшфлоу» прорабатывалась, но снята как преждевременная.

«Карта трат» прорабатывалась, но снята: Дзен-мани почти не отдаёт координаты для
операций с банковской синхронизации (push/SMS) — фича была бы пустой у большинства.

## 1.1 — добавление операций ✅

Версия **1.1.0**: операции теперь можно **создавать** прямо в приложении (кнопка
«Добавить» на странице «Операции», режим API). Любой тип — расход/доход/возврат/
перевод, включая мультивалютный. Черновик сразу виден в списке с пометкой «не
синхронизировано» и уезжает в облако при следующем Push. Реализовано поверх
существующего push-конвейера (примитив создания «под свежим id» уже был обкатан в
восстановлении удалённых).

## 1.0 — выпущено ✅

Версия **1.0.0** выпущена 2026-06-12: весь объём «До 1.0» закрыт и обкатан.
Двусторонняя синхронизация — полностью; все пользовательские пожелания (бюджет
план/факт, FIRE с капиталом, регулярные, хэштеги, счета вне баланса) — внедрены.

### Двусторонняя синхронизация (push) — ГОТОВО ✅
Закрыт весь объём: дата/получатель/бренд/комментарий/сумма/валюта/категория,
смена типа (Расход↔Доход↔Возврат и на/с «Перевод»), смена счёта (в т.ч. счетов
перевода), мультивалютные операции (смена типа на FX-строках, перевод между
счетами разной валюты, перенос на счёт другой валюты, флип FX-операции в перевод —
op-суммы отбрасываются, ноги собираются в валютах счетов), плюс детект конфликтов
перед push. Осталась лишь ниша: редактирование уже существующего FX-перевода с
op-суммами — скипается с понятным сообщением (экзотика).
