# Roadmap

Живой список того, что хотим доделать. Не строгий план — приоритеты двигаем по
ситуации. Статусы выверены по коду (а не по комментариям), дата ревизии: 2026-06-09.

## До 1.0

### Двусторонняя синхронизация (push) — закрыть оставшиеся пробелы
Что уже отправляется в облако: дата, получатель, бренд, комментарий, сумма,
валюта, категория/подкатегория и **смена типа** Расход ↔ Доход ↔ Возврат.
Остаётся:

- [ ] **Смена типа на/с «Перевод»** — сейчас пропускается (нужен второй счёт).
- [ ] **Мультивалютные операции** при смене типа — пропускаются (операционные
      суммы в валюте, отличной от счёта).
- [ ] **Смена счёта операции** — применяется локально, но **в облако не пушится**
      (`zenmoneyPush.ts` скипает с сообщением «Phase 1 не поддерживает…»).
- [ ] **Детект конфликтов при push** (бывшая «Phase 1.1») — сейчас сервер решает
      «последний выиграл» по `changed`; хотим предупреждать о расхождениях.

### Качество / надёжность
- [ ] **Юнит-тесты на фильтрацию** — `applyFilters` (период/категория/счёт/валюта/
      поиск/пересечения) проверялась только вручную в браузере. Покрыть тестами.
- [ ] **Аудит тултипов в тёмной теме** — починили график «Категории»; проверить
      остальные одно-серийные графики (Top, Sankey, бары Счетов) на читаемость
      текста подсказки.

### Документация (быстрые правки, доки устарели)
- [ ] **README + диалог подтверждения в ImportPage**: всё ещё пишут, что «смена
      типа операции не передаётся» — устарело, смена типа (кроме Перевода и
      мультивалюты) уже отправляется. Привести к формулировке из `HelpPage`.
- [ ] **Требования к Node в README**: указано «18+ (рекомендуется 20+)», а Vite 8
      и Dockerfile уже на Node 26 — обновить.

## После 1.0 / идеи

- [ ] **Cashflow / Trends — мягкий дефолт периода.** Это графики «по месяцам/по
      истории», но при заходе показывают текущий месяц (глобальный фильтр). Не
      баг, но можно дать им свой период-селектор (как в Календаре), чтобы на
      старте показывалась осмысленная история.
